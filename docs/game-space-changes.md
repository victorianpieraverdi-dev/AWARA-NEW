## 3. Мобильный слой опыта вселенной

### Что добавлено

В `index.html` есть:

- кнопка `ОПЫТ`;
- панель `mobile-universe-experience-panel`;
- блоки `Свет`, `Уровень`, `Луч`, `Зодиак`;
- быстрые переходы к панелям `Прогресс`, `Квесты`, `Ключи`, `Лила`;
- текстовый блок, объясняющий, что сейчас вынесено в галактику, а остальное хранится в опыте.

Ключевой смысл панели:

```text
Активный знак, активный луч, Ра и одна сфера видимы.
Остальной опыт хранится в панелях.
```

### Как работает обновление данных

В `index.html` подключён модуль:

```js
import { createUniverseSnapshot } from './js/universe/universeProgression.js';
```

Мобильный фокус и панель опыта обновляются через снимок состояния игрока:

```js
var snapshot = createUniverseSnapshot(getPlayerState());
```

Панель получает:

- количество света;
- текущий уровень игрока;
- активный знак зодиака;
- активный луч;
- описание того, что сейчас вынесено в галактику.

### Стили мобильного опыта

В `css/main.css` есть:

- стили кнопки `#mobile-universe-experience-btn`;
- стили панели `#mobile-universe-experience-panel`;
- состояние `.open`;
- оформление заголовка, сетки статов, списка кнопок и заметки.

Панель появляется только в мобильных режимах `universe-mobile-overview` и `universe-mobile-focus`.

## 4. Новый принцип мобильной вселенной

Зафиксирован принцип:

```text
everything is stored
not everything is displayed
the player chooses what becomes visible
```

Практический перевод для AWARA:

- игра не удаляет глубину;
- игрок не обязан видеть всю глубину сразу;
- весь опыт, карты, ключи, лучи, знаки, сферы, храмы и связи хранятся в структуре игрока;
- мобильный экран показывает только выбранный активный срез;
- будущий UI должен дать игроку возможность вручную закреплять объекты в галактике и убирать их обратно.

Это важный архитектурный поворот: телефон становится не уменьшенной копией desktop-космоса, а медитативным фокус-экраном.

## 5. Уровни эволюции игрока

В проекте есть 6 уровней:

```text
1. Ahamkara / Mind
2. Soul / Feeling
3. Jiva / Earth and Star Temple
4. Spirit / Cosmos and Service
5. Sobor / Galactic and Universal Architecture
6. Divine Board Game
```

В коде они представлены в `js/universe/universeProgression.js` как `UNIVERSE_LEVELS`.

Каждый уровень хранит:

- `id`;
- порядок;
- название;
- подзаголовок;
- минимальный свет;
- цвет;
- иконку;
- фокус;
- gameplay-формулу;
- unlocks;
- глубину карт;
- открытые generation layers.

### Пороговая модель света

| Уровень | Минимальный свет | Смысл |
|---|---:|---|
| Ахамкара | 0 | ум, культура, ключи, первые карты |
| Душа | 500 | тело, чувство, тигель, дневник состояния |
| Джива | 2000 | земля, храм, пространство, Васту |

## 6. Карты, ключи, редкость и глубина

Архитектурно закреплено: карты не должны быть только случайным коллекционным предметом.

Карты становятся кристаллизованным опытом игрока.

На выдачу карты должны влиять:

- уровень игрока;
- текущая матрица восприятия;
- натальные данные;
- дневной календарь;
- текст Тигеля;
- активированные элементы;
- активированные человеческие принципы;
- привлечённые агенты;
- мифологический резонанс;
- выполненные квесты;
- количество света;
- кооперация;
- подтверждённые медиа.

Возможные результаты:

- карта;
- фрагмент карты;
- ключ;
- артефакт;
- резонанс агента;
- резонанс матрицы;
- материал для Звёздного Храма;
- космические врата;
- ход в Игре Мироздания.

Главный принцип редкости: высокая карта приходит чаще только тогда, когда игрок способен нести соответствующую глубину, свет и ответственность.

## 7. Свет, Светкоин и светообмен

В документации разделены два слоя:

### Внутренний свет

- рост души;
- не должен продаваться или отдаваться;
- отражает развитие игрока;
- связан с искренностью, практикой, служением, творчеством и интеграцией тени.

### Светкоин

- обменная форма сгущённого света;
- может начисляться за подтверждённый вклад;
- может поддерживать услуги, дары, обмен картами, социальные действия и совместное созидание.

Важно: это не просто рынок. По видению Pavel'а светообмен должен быть способом заявить:

```text
что я могу дать
что я ищу
с кем хочу соединиться
какое действие готов воплотить в реальность
```

То есть светообмен связывает игру с планетой Земля, реальными встречами, созидательными проектами, путешествиями, практиками и служением.

## 8. Сиддхи, аура, тороид и сила игрока

В архитектуре описано, что по мере роста света у игрока усиливается его поле:

- аура;
- электромагнитный тороид;
- способность притягивать события и уроки;
- способность взаимодействовать с силами, мирами и локами;
- способность строить свои пространства, артефакты, храмы и миры.

Игровой перевод:

- больше света усиливает поле игрока;
- поле влияет на качество встреч, заданий, карт и резонансов;
- высокие уровни открывают локи, космические энергии, соборные действия и будущую настольную карту мироздания;
- сиддхи не должны быть механикой власти, а должны выражать зрелость, ответственность и служение.

## 9. Генерации по уровням

Архитектурно зафиксирована постепенная генерация образов и медиа:

| Уровень | Что может генерироваться |
|---|---|
| Ахамкара | простая картинка ума / культурного ключа |
| Душа | образ чувства, тела, Тигеля, внутреннего состояния |
| Джива | земля, храм, Васту-пространство, дом |
| Дух | космос, род, служение, планетарная связь |
| Собор | локи, соборы, фестивальные поля, высшие карты |
| Игра Мироздания | полный настольный вид мира, квесты, карты, союзы, светообмен |

Принцип:

```text
чем глубже внутренняя архитектура игрока,
тем проще и чище внешний интерфейс,
но тем больше слоёв может быть раскрыто по выбору.
```

## 13. Что ещё не завершено

### 13.1. Pin / unpin объектов в галактике

Принцип есть, но полного UI пока нет.

Нужно добавить:

- список объектов, доступных для закрепления;
- действие `закрепить в галактике`;
- действие `убрать из галактики`;
- сохранение выбора в player state;
- ограничение плотности, чтобы телефон не перегружался.

### 13.2. Полный светообмен

Сейчас это архитектурная идея. Для реализации нужны:

- профиль того, что игрок предлагает;
- профиль того, что игрок ищет;
- заявки и отклики;
- связь с реальными действиями;
- правила безопасности;
- backend или другой общий слой данных.

### 13.3. Divine Board Game

Идея описана, но верхний игровой слой ещё впереди.

Нужно отдельно проектировать:

- карту поля;
- правила ходов;
- карты и колоды;
- связь с личной историей игрока;
- коллективные квесты;
- условия входа на верхний уровень.

### 13.4. README и канон

Найдено расхождение, которое стоит разобрать отдельной задачей:

- в правилах проекта фигурирует канон 33 матрицы;
- в `docs/roadmap.md` есть места с 32 матрицами;
- README может быть устаревшим относительно количества экранов.

Этот документ не меняет канон. Он только фиксирует, что требуется отдельная синхронизация справочных документов.
